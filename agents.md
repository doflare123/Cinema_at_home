# AGENTS

## Контекст проекта
Проект «Кино-клуб» — закрытая платформа для друзей, заменяющая Excel и Telegram-бота как источник данных и операционную систему процессов: ожидания, недельные голосования, пост-просмотровые оценки, комментарии, статистика.

## Целевое состояние (Desired Outcome)
1. Сайт становится единственным источником правды после одноразового импорта Excel.
2. Публичная часть остается read-only, закрытая часть обслуживает все действия участников и админов.
3. Критическая бизнес-логика живет в backend, Telegram используется как auth/notification/entrypoint слой.
4. Доменная модель поддерживает: одиночные фильмы, франшизы, ожидание, недельные паки, review-режимы, статистику.
5. MVP поставляются поэтапно без смешивания ответственности между слоями.

## Декомпозиция задачи из movie_club_project_spec.md

### 1) Доступ, роли, авторизация
Цель:
- Разграничить публичный и закрытый контуры.
- Внедрить Telegram-авторизацию с модерацией новых пользователей.

Ожидаемый результат:
- Роли: `member`, `admin`.
- Статусы: `pending`, `active`, `rejected`, `blocked`.
- Новый пользователь попадает в `pending` до решения админа.
- Гость видит данные, но не может менять их.

Критерии готовности:
- Backend валидирует права доступа ко всем защищенным endpoint.
- Telegram auth data проверяется на backend.

### 2) Домен фильмов и франшиз
Цель:
- Поддержать отдельные фильмы и объединение фильмов в франшизы.

Ожидаемый результат:
- Фильм — отдельная сущность с собственной пост-просмотровой оценкой.
- Франшиза — отдельная сущность над фильмами.
- Ожидание для франшизы общее, пост-оценки — у каждого фильма отдельно.

Критерии готовности:
- Нельзя моделировать франшизу как «фильм».
- Связи фильм↔франшиза сохраняют порядок/часть.

### 3) Система оценок ожидания
Цель:
- Реализовать входной фильтр интереса к просмотру.

Ожидаемый результат:
- Голос ожидания: `1..10` или `-` (refuse).
- `-` не участвует в среднем.
- Проходной порог: `expectation_avg >= 5.0`.
- Админ не может вручную обойти порог.

Критерии готовности:
- Валидация типов голосов на backend.
- Корректная агрегация среднего только по числовым значениям.

### 4) Недельные паки и голосование
Цель:
- Организовать отдельный цикл недельного голосования для отбора в внешнее колесо.

Ожидаемый результат:
- Сущности пака: `draft`, `voting`, `closed`, `archived`.
- Шкала голоса: `+3, +2, +1, 0, -2`.
- Лимиты на пользователя в одном паке:
  - `+3` — 1 раз
  - `+2` — 2 раза
  - `+1` — 2 раза
  - `-2` — 1 раз
  - `0` — без лимита
- Итоги считаются как сумма голосов по фильму.

Критерии готовности:
- Backend блокирует нарушения лимитов.
- UI/API отдает агрегаты и детализацию "кто как голосовал".

### 5) Оценка после просмотра (review)
Цель:
- Поддержать два режима финальной оценки фильма.

Ожидаемый результат:
- Простой режим: целое `1..10` + необязательный комментарий.
- Режим «смешарика»: оценки по критериям, итог = среднее, ручная правка итога запрещена.
- Комментарии публичны.

Критерии готовности:
- Ограничение режимов и диапазонов оценок на backend.
- Идемпотентное обновление оценки пользователя для фильма.

### 6) Предложения фильмов и модерация
Цель:
- Дать участникам предлагать фильмы, сохраняя контроль качества через админа.

Ожидаемый результат:
- Предложение фильма проходит статусы `pending/approved/rejected`.
- При approved фильм попадает в основной каталог.

Критерии готовности:
- Аудит: кто предложил, кто и когда модерировал.

### 7) Интеграция с Кинопоиском
Цель:
- Ускорить и стандартизировать добавление фильмов.

Ожидаемый результат:
- Поиск через API, выбор результата, автозаполнение карточки.
- Ручная анкета при отсутствии фильма в API.

Критерии готовности:
- Backend отделяет source: `kinopoisk` / `manual`.
- Ошибки API не ломают ручной сценарий.

### 8) Telegram-бот и Mini App
Цель:
- Сделать Telegram интерфейсом доступа и уведомлений, но не местом бизнес-логики.

Ожидаемый результат:
- Бот уведомляет, отправляет итоги, сообщает о заявках/предложениях.
- Mini App использует тот же backend, что и основной сайт.

Критерии готовности:
- Нет критических вычислений/ограничений в боте.
- Голоса и оценки валидируются только backend-сервисом.

### 9) Публичные страницы и аналитика
Цель:
- Обеспечить прозрачный просмотр истории клуба и метрик.

Ожидаемый результат:
- Публичные карточки фильмов/франшиз, оценки, комментарии.
- Статистика по фильмам, франшизам, участникам, жанрам, пакам, времени.

Критерии готовности:
- Метрики считаются из сырых голосов/оценок, а не ручных агрегатов.

### 10) Архитектура и инфраструктура
Цель:
- Получить модульный backend и предсказуемый deployment.

Ожидаемый результат:
- Backend-модули: `/auth`, `/users`, `/movies`, `/franchises`, `/expectations`, `/reviews`, `/weekly-packs`, `/proposals`, `/statistics`, `/telegram`, `/kinopoisk`, `/admin`, `/import`.
- Стек: Go + (Gin) + PostgreSQL (+ GORM/sqlc), frontend на Next.js + Tailwind.
- Запуск через Docker Compose.

Критерии готовности:
- Вся критичная валидация в backend.
- Минимум дублирования бизнес-логики между вебом и Telegram.

## Roadmap (MVP-порядок)
1. MVP1: Замена Excel и базовый контур пользователей/оценок/админки.
2. MVP2: Интеграция Кинопоиска.
3. MVP3: Франшизы.
4. MVP4: Недельные паки + лимиты + Telegram-итоги.
5. MVP5: Telegram Mini App.
6. MVP6: Расширенная аналитика.

## Постоянные правила выполнения задач (обязательные)
1. Для сложных задач почти всегда использовать субагентов и подбирать их по типу работ:
- структура проекта/поиск границ — `mapper`;
- SQL/схемы/запросы — `sql-pro`;
- backend/Go — `backend-developer` или `golang-pro`;
- API-контракты и документация — `api-designer` / `api-documenter`;
- архитектурные решения — `architect-reviewer`.

2. После каждой завершенной задачи обязательно запускать отдельного `reviewer`, который проверяет diff, недоделки, баги, неточности и регрессы.

3. Любые написанные изменения в коде обкладывать unit-тестами и всегда запускать эти тесты.

4. Всегда чистить кэш (в рамках безопасного workspace-пути) перед финальной верификацией сборки/тестов, если это применимо к задаче.

5. Если вносятся non-ASCII тексты, обязательно проверять, что текст не битый (кодировка/отображение).

6. Если что-то непонятно или данных недостаточно, задавать уточняющий вопрос. Ничего не придумывать и явно говорить о неопределенности.

## Принципы качества
- Не смешивать ожидание и недельное голосование.
- Не переносить бизнес-логику во frontend/бот.
- Проверять ограничения и инварианты на backend.
- Двигаться инкрементально по MVP-плану с проверяемыми критериями готовности.
